\chapter{Metodología}
\label{chap:metodologia}

En este capítulo se describe el diseño del sistema, desde la formulación del problema de aprendizaje supervisado hasta el marco de evaluación económica. La metodología se ha diseñado para garantizar la robustez estadística y la aplicabilidad de los resultados en un entorno operativo real.

\section{Formulación del Problema}
El objetivo es predecir la demanda acumulada de un producto en una tienda específica durante un horizonte de planificación determinado ($H$). Formalmente, definimos la demanda observada para una serie $s$ en el tiempo $t$ como $y_{s,t}$. El objetivo (\textit{target}) de nuestro modelo en el instante $t$ es la demanda total durante el \textit{lead time}:

\begin{equation}
    Y_{s,t} = \sum_{i=0}^{H-1} y_{s, t+i}
\end{equation}

Esta formulación permite que el pronóstico esté directamente alineado con la decisión de inventario: cuántas unidades se necesitan para cubrir el periodo hasta la próxima reposición.

En la implementación, esta variable se construye desplazando la demanda observada hacia el futuro dentro de cada serie tienda-producto. Para cada origen temporal $t$, se agregan los valores $y_{s,t}$, $y_{s,t+1}$, ..., $y_{s,t+H-1}$, sin mezclar observaciones de distintas series. Por tanto, la demanda observada diaria no se usa como objetivo aislado, sino como componente de una etiqueta acumulada de \textit{lead time}. Las últimas observaciones de cada serie se descartan cuando no existe un horizonte futuro completo para calcular esta suma.

Esta distinción es crítica para evitar \textit{data leakage}: los valores futuros se usan únicamente para construir la etiqueta durante entrenamiento y evaluación, mientras que las variables explicativas se calculan con información disponible antes del origen de predicción, mediante retardos positivos y ventanas móviles históricas.

\section{Ingeniería de Características y Política Anti-Leakage}
Para capturar la complejidad de la demanda en \textit{retail}, se construye un panel de datos enriquecido con diversas familias de características. Un principio fundamental en el diseño es la \textbf{prevención del sesgo de información futura} (\textit{data leakage}).

\begin{itemize}
    \item \textbf{Características Temporales}: Componentes cíclicos extraídos de la fecha, como el día de la semana, semana del año, mes y indicadores de festivos.
    \item \textbf{Variables de Retardo (\textit{Lags})}: Valores históricos de la demanda, descuentos y horas de rotura de stock. Se utilizan retardos positivos para asegurar que el modelo solo utiliza información disponible en el momento de la predicción.
    \item \textbf{Estadísticas Móviles (\textit{Rolling Windows})}: Medias, sumas y desviaciones típicas calculadas sobre ventanas deslizantes (ej. 7 y 28 días) para capturar tendencias y volatilidad reciente.
    \item \textbf{Variables Exógenas}: Datos meteorológicos (temperatura, precipitación, humedad) y variables contextuales (actividad promocional).
    \item \textbf{Identificadores Estáticos}: Variables categóricas que representan la jerarquía del producto y la ubicación de la tienda.
\end{itemize}

Los retardos y ventanas concretos se parametrizan desde la configuración del experimento, lo que permite comparar distintas granularidades históricas sin alterar el código. En la configuración base se utilizan retardos de 1, 7, 14 y 28 días, junto con ventanas móviles de 7 y 28 días, capturando tanto memoria de muy corto plazo como patrones semanales.

Después de construir estas variables, se eliminan las filas que no disponen de historial suficiente para calcular los retardos o ventanas requeridas. Esta eliminación se realiza de forma independiente por \texttt{series\_id}. Por ejemplo, con un retardo máximo de 28 días y una ventana móvil máxima de 28 días, las primeras 28 observaciones de cada serie tienda-producto no pueden utilizarse como filas supervisadas, ya que todavía no existe historial completo hasta $t-1$. También se eliminan las filas sin \texttt{target\_lead\_time\_demand}, normalmente situadas al final de cada serie por falta de horizonte futuro completo.

En la configuración base, el horizonte de predicción es $H=7$. Por tanto, además de las primeras 28 observaciones descartadas por construcción de variables históricas, las últimas $H-1=6$ observaciones de cada serie no tienen etiqueta futura completa. Con el mínimo configurado de 70 días de historial por serie, el número mínimo teórico de orígenes supervisados utilizables es:

\begin{equation}
    70 - 28 - (7 - 1) = 36
\end{equation}

Este recorte temporal no constituye una pérdida arbitraria de datos, sino una consecuencia necesaria de formular un problema supervisado sin fuga de información futura. Finalmente, la columna objetivo queda excluida explícitamente del conjunto \texttt{feature\_columns}; de este modo, la matriz de entrada contiene solo información disponible en el origen de predicción, mientras que la demanda futura acumulada se conserva únicamente como etiqueta supervisada.

\section{Validación Walk-Forward}
Dada la naturaleza temporal de los datos, no es posible utilizar una validación cruzada aleatoria tradicional, ya que violaría la estructura causal del tiempo. En su lugar, se implementa un esquema de \textbf{validación temporal \textit{walk-forward}} con ventanas expansivas.

El proceso se divide en $N$ pliegues (\textit{folds}). Para cada pliegue:
\begin{enumerate}
    \item El conjunto de entrenamiento incluye todos los datos desde el inicio hasta una fecha de corte $T_{train}$.
    \item Se deja un \textit{gap} de seguridad igual al horizonte de predicción $H$ para evitar que información del futuro se filtre en las variables retardadas.
    \item El conjunto de validación comprende los datos en el intervalo $[T_{train} + H, T_{train} + H + \Delta]$, donde $\Delta$ es el tamaño de la ventana de validación.
\end{enumerate}

La construcción de los pliegues exige un número mínimo de fechas distintas en el panel. Si $D_0$ es el número de días iniciales de entrenamiento, $N$ el número de pliegues, $\Delta$ el tamaño de cada ventana de validación y $H$ el horizonte de predicción, el requisito operativo es:

\begin{equation}
    D_{min} = D_0 + N \cdot \Delta + H - 1
\end{equation}

El término asociado al horizonte garantiza que el último origen de validación tenga suficiente futuro observado para calcular la demanda acumulada del horizonte. En la configuración base, este requisito toma el valor $45 + 3 \cdot 7 + 7 - 1 = 72$ fechas distintas.

Operativamente, los pliegues se construyen sobre el calendario global de fechas únicas del panel, no sobre cada serie individual. El primer pliegue comienza tras $D_0$ fechas de entrenamiento inicial y cada pliegue posterior desplaza el inicio de validación $\Delta$ fechas hacia delante. Para el pliegue $j$, el índice inicial de validación es $D_0 + j \cdot \Delta$ y el índice final es $D_0 + (j+1)\cdot\Delta - 1$. El último índice admisible es $D - H$, donde $D$ es el número total de fechas únicas; de este modo, el último origen de validación aún dispone de $H$ observaciones futuras, incluyendo el propio día de origen, para construir la etiqueta de demanda acumulada.

La optimización de hiperparámetros sigue una política temporal coherente con este esquema. Antes del \textit{backtesting} oficial, el sistema ejecuta una única fase de sintonización con Optuna utilizando exclusivamente observaciones anteriores al final del entrenamiento del primer pliegue oficial. Dentro de ese subconjunto se aplica, a su vez, una partición temporal interna para medir el error de validación de cada prueba. Los hiperparámetros seleccionados quedan fijados después para todos los pliegues \textit{walk-forward}. Esta decisión evita fuga de información desde las ventanas oficiales de validación y reduce el coste computacional frente a una estrategia más estricta, pero mucho más cara, en la que se volvería a sintonizar el modelo dentro de cada pliegue.

Sobre este mismo flujo temporal, el sistema incorpora un mecanismo ligero de vigilancia adaptativa del rendimiento. Tras cada pliegue se monitoriza el MAE del modelo global de \textit{boosting} mediante un detector secuencial de Page-Hinkley parametrizado por un umbral, una tolerancia $\delta$ y un número mínimo de observaciones. Si la degradación acumulada del error supera el umbral configurado, el evento se registra como posible \textit{performance drift}. Cuando la configuración experimental activa esta opción, la alerta no modifica retrospectivamente el pliegue actual, pero sí fuerza el reentrenamiento del modelo en el siguiente pliegue, preservando la causalidad temporal del protocolo.

Este esquema garantiza que el modelo se evalúe siempre sobre datos que nunca ha visto y que son cronológicamente posteriores a su entrenamiento, simulando un despliegue real en producción.

\section{Cuantificación de la Incertidumbre y Calibración}
El sistema no genera únicamente una predicción puntual, sino un conjunto de cuantiles operativos ($\tau \in \{0.1, 0.5, 0.9\}$) para soportar la toma de decisiones bajo riesgo. Para asegurar que estos cuantiles sean estadísticamente válidos, se integra un marco de \textbf{Conformal Prediction}.

\subsection{El Problema de los Cuantiles Crudos}
Los modelos de \textit{Gradient Boosting} (como CatBoost o LightGBM) pueden entrenarse utilizando una función de pérdida de \textit{Pinball}, lo que les permite estimar cuantiles. Sin embargo, en la práctica, estos cuantiles suelen estar mal calibrados debido al \textit{overfitting} o a cambios en la distribución de los datos, lo que significa que un intervalo del $80\%$ nominal podría cubrir solo el $65\%$ de los datos reales.

\subsection{Marco de Conformal Prediction}
Para corregir este sesgo, se implementa una variante del método de \textit{Split Conformal Prediction}. El proceso se formaliza de la siguiente manera:

\begin{enumerate}
    \item \textbf{Corte de Calibración}: Se reserva un conjunto de datos de calibración ($\mathcal{D}_{cal}$) que no ha sido utilizado durante el entrenamiento del modelo. En este sistema, se utilizan los últimos 21 días del conjunto de entrenamiento para este fin.
    \item \textbf{Score de No-Conformidad}: Para cada observación en $\mathcal{D}_{cal}$, se calcula un \textit{score} de error que mide la discrepancia entre el valor real $y_i$ y el intervalo predicho. Cuando el modelo base ofrece cuantiles, se usa un score de tipo CQR basado en la máxima desviación respecto a los extremos del intervalo; cuando solo ofrece predicción puntual, se usa el residuo absoluto.
    \item \textbf{Ajuste de Cuantiles}: Se calcula el percentil corregido de los errores observados en el conjunto de calibración. El cuantil final para una nueva observación $x_{test}$ se define como:
    \begin{equation}
        q_{final} = \hat{q}_{\tau}(x_{test}) + Q(1-\alpha, \{E_i\}_{i \in \mathcal{D}_{cal}})
    \end{equation}
    donde $Q$ es la función de cuantil empírico sobre los errores de calibración.
\end{enumerate}

Este enfoque garantiza, bajo la asunción de intercambiabilidad de los datos, que el error de cobertura esté acotado teóricamente, proporcionando intervalos de confianza robustos que se ensanchan automáticamente en periodos de alta incertidumbre y se estrechan cuando el modelo es preciso.

\section{Marco de Evaluación Multi-Objetivo}
La evaluación del sistema se realiza en tres niveles complementarios:

\begin{itemize}
    \item \textbf{Métricas Predictivas}: MAE (\textit{Mean Absolute Error}) para medir la precisión central y RMSE (\textit{Root Mean Squared Error}) para penalizar grandes desviaciones.
    \item \textbf{Métricas Probabilísticas}: \textit{Pinball Loss} para evaluar la calidad de los cuantiles y \textit{Coverage} para verificar si el porcentaje de datos que caen dentro del intervalo de confianza coincide con el nivel de confianza teórico.
    \item \textbf{Métricas Económicas}: Se utiliza el modelo del \textit{Newsvendor} para calcular el coste total operativo:
    \begin{equation}
        Coste_{s,t} = (c_{o,s} \times Overstock_{s,t}) + (c_{u,s} \times Stockout_{s,t})
    \end{equation}
    donde $c_{o,s}$ es el coste unitario por exceso de stock de la serie $s$ y $c_{u,s}$ el coste de oportunidad por venta perdida para esa misma serie.
\end{itemize}

En la versión actual del sistema, estos costes no son constantes globales para todo el panel, sino perfiles sintéticos dependientes de la combinación tienda-producto. En particular, $c_{o,s}$ aumenta cuando una serie presenta mayor perecibilidad sintética o un patrón de baja rotación, mientras que $c_{u,s}$ aumenta cuando la serie muestra mayor intensidad de demanda o mayor tensión histórica de \textit{stockout}. De este modo, el fractil crítico del modelo del \textit{Newsvendor} deja de ser único para todo el sistema y pasa a adaptarse a la criticidad operativa de cada serie individual.
