\chapter{Metodología}
\label{chap:metodologia}

En este capítulo se describe el diseño del sistema, desde la formulación del problema de aprendizaje supervisado hasta el marco de evaluación económica y de Investigación de Operaciones (OR). La metodología se ha rediseñado para trascender la predicción puramente estadística, garantizando la robustez operativa y la resolución de restricciones logísticas reales.

\section{Formulación del Problema}
El objetivo operativo es determinar la cantidad de pedido que debe emitir el responsable de reposición para cada SKU y cada fecha de decisión. Para ello, primero se estima la demanda acumulada de un producto en una tienda específica durante un horizonte de planificación determinado ($H$). Formalmente, definimos la demanda observada para una serie $s$ en el tiempo $t$ como $y_{s,t}$. El objetivo (\textit{target}) de nuestro modelo en el instante $t$ es la demanda total durante el \textit{lead time}, que actúa como entrada para la política de aprovisionamiento:

\begin{equation}
    Y_{s,t} = \sum_{i=0}^{H-1} y_{s, t+i}
\end{equation}

Esta formulación alinea el pronóstico con la decisión de inventario: cuántas unidades se necesitan para cubrir el periodo hasta la próxima reposición y, por tanto, qué cantidad debería pedirse mañana para minimizar el coste operativo. Para evitar la fuga de información futura (\textit{data leakage}), los valores futuros se usan únicamente para construir esta etiqueta durante el entrenamiento, mientras que las variables explicativas se calculan exclusivamente con información disponible antes del origen de predicción.

\section{Optimización Multi-Objetivo y Frente de Pareto}
Históricamente, los modelos de \textit{demand forecasting} minimizaban una única métrica de error promedio, como el MAE o el RMSE. Sin embargo, en un problema de optimización de inventarios donde la gestión de la incertidumbre es vital, buscar únicamente reducir la media del error puede resultar en modelos con varianza infraestimada o sobreajustada.

Para abordar este problema, el sistema implementa una **Búsqueda de Hiperparámetros Multi-Objetivo** utilizando \textit{Optuna}. Durante la fase de sintonización, el algoritmo evolutivo (basado en NSGA-II) optimiza simultáneamente dos funciones de pérdida en conflicto:

\begin{enumerate}
    \item \textbf{Precisión Puntual (MAE)}: Penaliza la magnitud absoluta del error de la predicción central. Representa la eficiencia a largo plazo del modelo.
    \item \textbf{Nitidez Probabilística (Winkler Score)}: Penaliza simultáneamente la amplitud de los intervalos de confianza predichos (penalizando intervalos vagos o excesivamente anchos) y la penalización asimétrica si el valor real cae fuera del intervalo (asegurando cobertura empírica).
\end{enumerate}

El resultado no es una única configuración de hiperparámetros "óptima", sino un **Frente de Pareto** donde cada punto representa un compromiso inmejorable entre la precisión económica esperada y la fiabilidad de las cotas de incertidumbre estimadas. El orquestador selecciona automáticamente la configuración más equilibrada antes de iniciar el \textit{backtesting} global.

\section{Cuantificación de la Incertidumbre: Mondrian Conformal Prediction}
Para soportar la toma de decisiones bajo riesgo, el sistema requiere garantías matemáticas sobre los intervalos de confianza generados. Aunque técnicas previas como el \textit{Split Conformal Prediction} ofrecen garantías de cobertura \textit{marginal} (ej. 80\% de acierto en media para todo el conjunto de datos), esta propiedad suele enmascarar desigualdades críticas en \textit{retail}: el modelo podría estar acertando el 95\% de las veces en productos estables y fallando estrepitosamente en productos intermitentes o estacionales.

Para solucionar este sesgo estructural, se ha implementado \textbf{Mondrian Conformal Prediction}. Este marco extiende la calibración de manera condicional:

\begin{enumerate}
    \item \textbf{Partición Taxonómica}: Las series temporales se agrupan en categorías homogéneas (ej. mediante su identificador categórico de jerarquía \texttt{third\_category\_id}).
    \item \textbf{Score de No-Conformidad Local}: En lugar de calcular una única cota de corrección ($q\_hat$) global, el sistema calcula una cota independiente para cada grupo en el conjunto de calibración.
    \item \textbf{Ajuste Condicional de Cuantiles}: El cuantil ajustado final garantiza que el error de cobertura esté acotado teóricamente dentro de \textit{cada categoría}, asegurando que tanto los productos de alta rotación (cuya incertidumbre es baja) como los lentos y ruidosos reciben el nivel de servicio deseado, ensanchando o contrayendo los intervalos de forma justa y localizada.
\end{enumerate}

\section{El Gemelo Digital Logístico: Simulación Dinámica (s, S)}
La evaluación tradicional en forecasting asume modelos estáticos (o de un solo periodo), como el modelo básico del \textit{Newsvendor}. Sin embargo, este supuesto ignora el hecho físico de que el inventario no vendido un día permanece en la estantería al día siguiente.

El proyecto rompe con esta limitación mediante el desarrollo de una \textbf{Simulación Dinámica de Política Order-Up-To}. Para cada \textit{fold} de la validación cruzada temporal, el simulador mantiene el estado físico de cada SKU, incluyendo:
\begin{itemize}
    \item \textbf{Stock Remanente (\textit{On-Hand})}: Unidades físicas disponibles.
    \item \textbf{Pedidos en Curso (\textit{On-Order})}: Unidades compradas que aún no han llegado debido al \textit{lead time}.
    \item \textbf{Demanda Diferida (\textit{Backlog})}: Unidades demandadas que no se pudieron servir y generan penalización o pérdida de fidelidad.
\end{itemize}

Bajo esta simulación, la orden sugerida $Q$ se ajusta restando la \textit{Posición del Inventario} actual del nivel óptimo sugerido por el cuantil del pronóstico probabilístico $S$. De este modo, la evaluación de costes operativos en la memoria captura fielmente el impacto devastador a largo plazo de errores correlacionados y del "efecto látigo".

\section{Simulación Operacional en Streaming: Evaluación de Cadencias de Reentrenamiento}
\label{sec:simulate_ops}

La validación cruzada temporal (\textit{walk-forward backtesting}) estima el rendimiento del modelo en condiciones experimentales controladas. Sin embargo, en producción surge una pregunta adicional de naturaleza MLOps: \emph{¿con qué frecuencia debe reentrenarse el modelo a medida que llegan datos nuevos?} Reentrenar demasiado frecuentemente supone un coste computacional elevado; hacerlo demasiado esporádicamente puede degradar las predicciones conforme el modelo queda desactualizado respecto a los patrones recientes.

Para responder esta pregunta de forma empírica sobre datos genuinamente no vistos, el sistema implementa una \textbf{Simulación Operacional en Streaming} sobre el split de evaluación. El procedimiento es el siguiente:

\begin{enumerate}
    \item Se entrena un modelo \textit{champion} inicial exclusivamente sobre el split de entrenamiento. Este modelo representa el estado del sistema en el día de despliegue, antes de que llegue ningún dato del periodo de evaluación.
    \item Se define un conjunto de \textbf{cadencias de reentrenamiento} $K = \{k_1, k_2, \ldots, k_n, \text{nunca}\}$, donde cada $k_i$ representa el número de días entre dos reentrenamientos consecutivos. La cadencia \textit{nunca} sirve como \textit{baseline}: el modelo inicial nunca se actualiza.
    \item Se itera día a día sobre el split de evaluación. Para cada fecha de decisión $t$ y cada cadencia $k$:
    \begin{itemize}
        \item Se construye la \textbf{historia disponible} $\mathcal{H}_t = \text{train} \cup \text{eval}[\text{date} < t]$, garantizando ausencia de \textit{leakage}.
        \item Si $(t_{\text{index}} + 1) \bmod k = 0$, se reentrena el modelo de esa cadencia sobre $\mathcal{H}_t$, incluyendo recalibración conformal.
        \item El modelo genera predicciones para el horizonte $[t, t+H-1]$.
        \item Se revelan los datos reales del periodo y se calcula el coste de inventario realizado mediante el modelo Newsvendor.
    \end{itemize}
    \item Al finalizar el streaming, se dispone de curvas de \textbf{coste acumulado} por cadencia sobre el mismo periodo y los mismos datos, lo que permite una comparación directa y equitativa.
\end{enumerate}

Los artefactos generados incluyen \texttt{cadence\_summary.csv} con métricas agregadas por cadencia (coste total, nivel de servicio, \textit{fill rate}, número de reentrenamientos) y \texttt{cumulative\_cost.png} con la evolución temporal del coste para cada política. Esta simulación constituye la evidencia empírica para justificar la política de reentrenamiento operacional del sistema.

\section{Resolución de Conflictos: Optimización Global por Programación Lineal}
Las recomendaciones individuales de cada SKU pueden ser matemáticamente óptimas a nivel aislado, pero el despliegue en un supermercado real está sujeto a severas restricciones de volumen físico de almacenaje o de capital presupuestario.

Para cerrar la brecha entre Machine Learning e Investigación de Operaciones (OR), la decisión final se orquesta a través de un \textbf{Módulo de Programación Lineal (LP)}. El problema se plantea como un \textit{Continuous Knapsack}:

\begin{itemize}
    \item \textbf{Variables de Decisión}: Cantidad a pedir de cada SKU.
    \item \textbf{Función Objetivo}: Maximizar la utilidad marginal (evitar el coste de oportunidad por rotura, fuertemente penalizado por la criticidad del producto).
    \item \textbf{Restricciones}: La suma total de unidades pedidas en el catálogo no debe exceder una constante $C_{max}$ (ej. Límite del camión logístico).
\end{itemize}

El motor utiliza el \textit{solver} moderno \texttt{highs} de \textit{SciPy} para recortar inventario inteligentemente allí donde el impacto económico de la rotura sea menor, dotando al sistema de total capacidad de actuación bajo estrés operativo real.
