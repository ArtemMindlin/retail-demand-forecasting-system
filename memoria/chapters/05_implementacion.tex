\chapter{Implementación y Arquitectura MLOps}
\label{chap:implementacion}

Este capítulo detalla la realización técnica del sistema. A diferencia de implementaciones puramente académicas en formato de *notebooks*, este proyecto se ha desarrollado como un \textbf{Sistema Inteligente End-to-End} bajo los más estrictos estándares de Ingeniería de Software y MLOps, listo para un entorno de producción industrial.

\section{Arquitectura del Sistema}
La estructura del código se organiza en torno a micro-arquitecturas especializadas:

\begin{enumerate}
    \item \textbf{Data Engineering y Quality Gates}: Encargado de la ingesta desde fuentes externas (Hugging Face) y la limpieza. Se ha implementado un estricto \textit{Data Quality Gate} que aborta la ejecución si detecta anomalías operativas (ej. datos obsoletos o duplicados) antes de contaminar los modelos.
    \item \textbf{Pipeline de Entrenamiento}: Gestiona la ingeniería de características sin \textit{leakage}, la optimización bayesiana multi-objetivo y el calibrado \textit{Mondrian Conformal}.
    \item \textbf{Decisión e Investigación de Operaciones}: Transforma los cuantiles en decisiones de reposición mediante la simulación de inventario (s, S) y la programación lineal.
\end{enumerate}

\section{Ecosistema Tecnológico}
El \textit{stack} tecnológico se ha seleccionado para maximizar la robustez, escalabilidad y explicabilidad:
\begin{itemize}
    \item \textbf{Motor de Modelado}: \texttt{CatBoost} y \texttt{LightGBM} para \textit{Gradient Boosting} probabilístico.
    \item \textbf{Optimización y OR}: \texttt{Optuna} (NSGA-II) para el frente de Pareto y \texttt{SciPy} (método \textit{highs}) para el \textit{Continuous Knapsack}.
    \item \textbf{Experiment Tracking (MLflow)}: El registro local de artefactos se ha migrado a un servidor \texttt{MLflow}. El sistema inyecta automáticamente todos los hiperparámetros, métricas (Winkler, PICP, costes) y gráficos generados, asegurando la reproducibilidad absoluta y la comparabilidad visual de cientos de ejecuciones.
    \item \textbf{Despliegue y Microservicios (FastAPI)}: El modelo no es un ente pasivo. Se ha implementado un microservicio asíncrono con \texttt{FastAPI} que expone el \textit{endpoint} \texttt{/predict\_orders}. Este servicio está diseñado para ser consumido directamente por el ERP de la cadena de suministro, integrando la inferencia y la optimización logística en tiempo real.
    \item \textbf{Dashboard Interactivo (Streamlit)}: Para acercar la matemática al tomador de decisiones, se ha desarrollado un \textit{What-If Dashboard}. Esta interfaz permite a los directores ajustar los costes de rotura y exceso o las capacidades de almacén de forma visual, re-simulando y comparando las decisiones logísticas al instante.
\end{itemize}

\section{Modos de Ejecución y Ciclo Operacional}
\label{sec:run_modes}

El sistema distingue explícitamente entre operaciones de investigación y operaciones de producción mediante cuatro modos de ejecución configurables a través del parámetro \texttt{run\_mode}:

\begin{enumerate}
    \item \textbf{\texttt{experiment}}: Modo de investigación completo. Ejecuta el \textit{backtesting walk-forward} con múltiples \textit{folds} temporales, compara las estrategias de demanda observada y latente, selecciona el modelo \textit{champion} mediante \textit{promotion logic} y genera artefactos de análisis exhaustivos (\texttt{report.md}, \texttt{champion\_registry.json}, gráficos de Pareto). Es el modo utilizado para validar el sistema antes de su despliegue.

    \item \textbf{\texttt{retrain}}: Modo de refresco operacional. Entrena el modelo \textit{champion} sobre la totalidad de los datos disponibles sin realizar evaluación por \textit{folds}. Guarda el modelo en disco en formato \texttt{.pkl}. Se ejecuta periódicamente en producción para mantener el modelo actualizado con los datos más recientes.

    \item \textbf{\texttt{score\_daily}}: Modo de producción diario. Carga el modelo \textit{champion} guardado, construye el \textit{inference frame} con los datos más recientes y genera \texttt{reorder\_recommendations.csv}: una fila por par tienda-producto con la \texttt{order\_quantity} óptima para el horizonte $H$, los tres cuantiles calibrados (\texttt{q\_0.1}, \texttt{q\_0.5}, \texttt{q\_0.9}) y la fecha de decisión. Este fichero es el entregable que consumiría el sistema de compras o el responsable de reposición.

    \item \textbf{\texttt{simulate\_ops}}: Modo de investigación MLOps. Ejecuta la simulación operacional en \textit{streaming} descrita en la Sección~\ref{sec:simulate_ops}, comparando múltiples cadencias de reentrenamiento sobre el split de evaluación.
\end{enumerate}

\subsection{Política Champion/Challenger y Promotion Logic}

El sistema implementa el patrón estándar de MLOps para la gestión del ciclo de vida de modelos. El \textbf{champion} es el modelo vigente en producción, registrado en \texttt{champion\_registry.json}. Un \textbf{challenger} es un candidato nuevo entrenado con datos más recientes o con una configuración diferente.

La \textbf{promotion logic} define los criterios bajo los cuales el challenger puede reemplazar al champion. El challenger solo es promovido si supera al champion en coste total de inventario sobre el \textit{holdout} y mantiene el nivel de servicio por encima del umbral mínimo configurado. Esta política garantiza que el modelo en producción nunca se degrada por un reentrenamiento deficiente: el champion permanece activo hasta que un challenger lo derrota de forma inequívoca según los criterios de negocio, no únicamente según métricas estadísticas.

\section{Calidad de Software e Infraestructura CI/CD}
Para garantizar que la deuda técnica no limite el escalado del sistema, se han aplicado prácticas fundamentales de \textit{DevOps}:
\begin{itemize}
    \item \textbf{Contratos Estrictos (Pydantic)}: Toda la configuración y los metadatos transitan a través de esquemas validados y tipados estáticamente, rechazando configuraciones incoherentes o derivas de datos no controladas.
    \item \textbf{Contenerización (Docker)}: La API, el servidor MLflow y el Dashboard de Streamlit están orquestados conjuntamente mediante \texttt{docker-compose.yml}, garantizando que cualquier investigador pueda levantar la arquitectura completa con un solo comando en cualquier sistema operativo.
    \item \textbf{Integración Continua (GitHub Actions)}: El repositorio cuenta con un pipeline de CI (\texttt{ci.yml}) que, ante cada nuevo \textit{commit}, despliega un entorno virgen, formatea el código (Ruff) y ejecuta exhaustivamente la batería de más de 80 tests automatizados (\texttt{Pytest}), garantizando la integridad arquitectónica ininterrumpida.
\end{itemize}

\section{Explicabilidad y Diagnóstico Automático}
Un sistema de apoyo a la decisión en \textit{retail} pierde su valor si opera como una caja negra. Por ello, se han integrado módulos de observabilidad (\textit{Observability}):
\begin{itemize}
    \item \textbf{Explicabilidad Global (SHAP)}: El sistema genera valores de Shapley Aditivos (\texttt{shap}) para el modelo Campeón, documentando gráficamente el peso real de los \textit{lags}, las variables climáticas o las jerarquías en la predicción.
    \item \textbf{Diagnóstico Post-Mortem}: En lugar de requerir análisis manual tras un fallo logístico, el sistema escanea sus propios errores. Si un SKU genera sobrecostes excesivos en la simulación, un módulo automático redacta un diagnóstico en el reporte final, indicando si la pérdida se debió a un \textit{Concept Drift} (basado en el detector Page-Hinkley), \textit{Cold Start} (falta de historia) o alta intermitencia estructural. Esto transforma el sistema de un mero proyector numérico a un \textbf{agente autodiagnóstico}.
\end{itemize}
